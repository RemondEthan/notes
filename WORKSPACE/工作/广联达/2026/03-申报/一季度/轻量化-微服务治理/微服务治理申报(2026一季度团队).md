#### 1. 案例介绍
1. 背景：
	1. PMLead腰部客户"卖出去"目标驱动
		- 海南农垦和沈阳鼓风机两个腰部意向客户，服务器资源多高成为可群成单卡点。
	2. 工具链降本不降效目标驱动
		- 在25年轻量化治理中，通过升级JDK21以及Spring3实现了约<10%的资源降低，通过部署时服务共享webserver方式实现了交付目标。同时，在服务高峰期会有额外不稳定风险
	3. AI开发提效带来的机遇
		- 微服务治理是非常繁杂且代码量巨大的工作，耗时长，且难以同时处理新增需求。有AI Coding的加持，可有效缩减50%的编码时间。
		- 自动化测试组件(MS)引入，rest api接口可自动化覆盖
	4. 最终目标制定
		- 腰部客户交付云成本由12~20w元降至5w元。工具链资源分摊降至48G，包括微服务资源（20G），同步任务资源（4G），OLAP查询引擎资源（24G）
2. 微服务现状：
	- 数据工具链部门的4 个核心组件（数据治理、底座、数据分析、数据组件），原有 40+ 个微服务
	- 微服务相关组件：Eureka/Gateway/xxl-job，以及查询分析引擎Starrocks
	- 数据工具链底层大数据引擎组件：dolphin/kuyybi
3. 任务梳理
	1. 技术方案
		- 识别功能相近微服务，相似功能服务整合
		- 微服务发现机制调整，xxl-job组件整合(去除Eureka/Gateway，xxl-job合并)
		- 云原生100%改造(去除kuyybi，自实现k8s资源管理)
		- 将无状态 Web 服务合并部署，共享资源池
	2. 任务开发
		- 模块整合，依赖迁移，公共模块重构
		- 非大数据任务组件任务云原生改造(50%AI Coding)
		- scala -> Java改造(AI Coding)
	3. 资源方案压测验证
		- PML腰部客户压测环境验证
		- xmgl-gor环境压测验证
4. 轻量化成果产出
	1. 微服务数量优化
	2. 中间件依赖优化(去除Eureka/Gateway，xxl-job合并)
	3. web服务部署优化
	4. 整体成果：数据治理：23->5/24G->9G  大数据底座：13->5 / 23.8G->3.54G  数据组件： 3-> 2 /  10.2G->2.6G
#### 2.价值成果
1. 用户视角（客户价值）
	是PML腰部客户轻量化专项的最终实现，实现助力PML从“卖出去”到长期“新架构”的升级，达成私有化部署5w/年目标。
	1. 腰部客户：
		- 首次实现可部署：实现工具链全链路服务48G 即可部署"，突破零障碍，不再让资源成为成单卡点，突破销售关键路径。
		- 简化运维管理：服务数量减少 62.5%，日常运维工作量大幅降低
		- 停机时间更短：部署参数精简~10%，部署过程耗时更短，2h->30min
		- 广西路建升级：测试环境：59.1G ->   46.7C->    生成环境：80.1G->   55.8C->
	2. 标准版客户：
		- 性能提升：同等资源总量前提下，服务分配资源更高，高并发场景下稳定性提升
		- 多版本标准化：根据工具链最新版本形成了从小微客户-腰部客户-头部客户-公有云客户标准化部署模板，实现运维沟通0成本。
2. 自身视角
	实现数据工具链适应湖仓一体以及AI Natve升级改造，实现架构瘦身，进一步通过技术缩减成本。
	1. 技术路径探索：
		- 轻量化深入探索：在产线和业务平台内，对轻量化课题已经经过了多轮次的探讨，从部署是逻辑合并到JDK升级均未能达到立竿见影的效果。代码级别微服务治理是复杂且风险较高的任务，通过工具链的服务治理探索结合AI开发提效，可大幅缩短服务治理周期。形成可复制模式。
	2. 工具链技术升级：
		- 架构升级：通过抓住轻量化专项契机，将工具链服务进行架构升级为1套平台+k8s模式，实现100%云原生，适配以湖仓一体为主的新型数据架构，推进轻计算引擎落地。
#### 3.过程沉淀

---
2.1 核心目标：
- 解决部署难题：在 48G 内存限制下成功部署数据工具链全量功能
- 大幅降低资源占用：减少 CPU 和内存消耗，满足轻量化环境要求
- 保证功能完整性：在资源受限情况下，不牺牲核心功能和用户体验
2.2 关键挑战：
- 技术复杂度：40+ 个微服务涉及 4 个组件，依赖关系复杂
- 时间紧迫：市场拓展急需轻量化版本，交付压力大
- 资源约束：48G 内存是硬性限制，无法突破
- 质量要求：不能因资源优化而影响产品稳定性和性能
3.行动：
3.1 技术方案设计：
3.1.1. 深度服务合并：
   - 识别功能相近的微服务，进行同功能整合
   - 将无状态 Web 服务合并部署，共享资源池
   - 优化服务间调用关系，减少通信开销
3.1.2. 差异化资源配置：
   - 腰部版：极致轻量化，针对 48G 内存环境优化
   - 标准版：功能完整，性能优化，满足大客户高并发需求
3.1.3. 动态资源管理：
   - 根据实际负载情况，动态调整服务资源配额
   - 共享中间件实例，减少资源重复占用
3.2 实施路径：
- **第 1 周：需求分析，调研腰部客户实际环境和需求
- **第 2-3 周：方案设计，制定服务合并方案和两套资源配置
- **第 4-8 周：开发实施，完成 4 个组件的服务合并
- **第 9-12 周：测试验证，功能测试、性能测试、部署验证

3.3 组织保障：
- 成立专项攻坚团队，4 个组件核心骨干参与
- 建立周例会机制，及时协调资源和解决问题
- 采用敏捷开发模式，快速迭代和验证
